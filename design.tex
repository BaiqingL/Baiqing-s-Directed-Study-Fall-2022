\title{Re-configurable Data Stream Processor}
\author{Baiqing Lyu \\Boston University}

\begin{abstract}
This document lays out the inspiration for a re-configurable data stream processor. It is a general framework for data stream processing, which can be used to build a wide variety of data processing applications. The uniqueness of this framework is that it is designed to be re-configurable, and supports better scalability and fault tolerance for stateful applications compared to current industry standard solutions. We take this novel stream data processor and compare it to a baseline to see it's overhead of leverage a distributed enviroment.
\end{abstract}

\section{Introduction}

\IEEEPARstart{T}{he} current state of the art in data stream processing is software frameworks like Apache Flink, Apache Spark, and Apache Storm. These frameworks all require in some capacity a centralized component in its data pipeline. That often leads to a potential single point of failure or ease of congestion that creates bottlenecks during production. The idea is to create a stream processor that has a certain tradeoff which can be expressed of having a bit higher latency than systems like Apache Flink, but lower than Apache Spark, including better scalability than those frameworks in addition to having capabilities for re-configuration.

\section{Requirements}
Being able to scale up and down without downtime is a fundamental requirement for this data stream processor. In addition, the processor should be decentralized in its workers and enable fast failure recovery if a node were to suddenly disconnect. It would also need to deliver \textbf{at least once} processing guarantees when ingesting incoming data. Current scope is to focus on the feasibility of a data stream processor that can be used in a distributed environment.
\section{Goals}

\begin{enumerate}
\item Create MVP of a ring dataflow backend that allow nodes to come in and out of the data processor.

\item Give the ability to input data into the dataflow and have a sink that can recieve the output.

\item Develop a MVP of a data stream processor that can support easy map reduce operations (i.e. wordcount)
\end{enumerate}
\section{Software Design}

The dataflow network design is inspired by Chord, an implementation of a distributed hash table\ (DHT). Chord creates a ring structure for the nodes and provies the ability to map a given key to some node.

As the Chord protocol supports the operation of load balancing in addition to mapping keys to nodes and controlling the behavior when the nodes are down, it is very helpful as the baseline for the data stream processor. The usage of finger tables within Chord provides a robust way to handle arbritrary node failures. In addition, this can also handle the scale up and down of nodes within the processor.

The main purpose of Chord is very simple in its mission, \textit{provided a key, the protocol will return the node that is responsible for the key}. We use this property to back state operators so that when workers enter the ring or unexpectedly leaves, the state is still preserved and can be recovered.

\section{Modifications}
Using the DHT implementation specified by Chord, we will modify it to support the map reduce functionality. Instead of the traditional setting of value to a node, there will be new methods that act as operators. For example instead of \texttt{setValue(value)} and letting Chord handle the corresponding node, we will have \texttt{sumOperator(value)} and let Chord set the value, and instruct the node to perform the sum operation. Similarly, we could have custom operators to perform any actions needed, as it will be chained as a secondary set to after storing the data in their corresponding nodes.

\subsection{Ring Design}
The stream processor will maintain a ring of nodes. Each node contains a finger table containing up to $m$ entries, $m$ being the number of bits in the hash key. Each $i^{th}$ entry in the finger table of node $n$ contains the next node $successor$ which is the $((n + 2 ^ {i - 1})\ mod\ 2^m)$ node.

\subsection{Event routing}
Each time there is a new event from the data source, a node lookup event is distributed to a node in the ring in a round-robin fashion. The node lookup event contains information regarding the key of the event. Then, the node that recieves the event $e$ will respond to the source with their id so that the source can contact and send the event to the correct node. This is done to prevent bouncing the entire data event within the ring sturcture, which can induce high IO.


\subsection{Ring stablization}
To ensure that the ring is always up to date, some sort of stabilization mechanism must be run periodically in the background so that the finger tables of the nodes are always correct.

Similar to the original Chord protocol, when a new node joins, the following actions are performed:
\begin{enumerate}
\item Initialize node $n$ (the predecessor and the finger table).
\item Notify other nodes to update their predecessors and finger tables.
\item The new node takes over its responsible keys from its successor.
\end{enumerate}

The stabilization algorithm that runs checks for dead nodes via heartbeat signals. The heartbeats are sent out by each node at the frequency of events that arrive as each node looks up where the event should be routed. For its next to its first successor to check their status, if there is no response, the node that discovered the dead successor will promote its next live successor to take their original place, closing the gap. The node will also take the key space of the dead node, and redistribute it among all nodes within its finger table.

When a worker leaves gracefully, it will also contact its predecessor so that the node will know immediately and run the stabilization algorithm.

\section{Result sink}
The result sink is a new sink that will be used to store the results of the map reduce operation. Since
we want to minimize the need for network activity, all results will be fed into a single node. This node
will be known as the \texttt{sink node}, and will be responsible for storing the results. This node does
not participate within the ring, and can be seen as a separate entity that is solely responsible for accepting
results and perform de-duplication and aggregation. This part of design is currently in its infant stage, as
having a single node responsible for result inherently introduces risk of failure.

In addition to an independent result sink, there is also a per node independent sink that could be used
if the data does not need to be aggregated or de-duplicated. This would then resolve the single point of
failure issue and allow for a more robust sinking mechanism for the stream processor.

\section{Benchmark}

\subsection{Baseline}
To discover the overhead of how much using a distributed processor would entail, we design a simple word count benchmark and execute it on the same machine. The benchmark features the entire works of William Shakespeare's works multiplied several times, as to add processing time for the machine. The whole text file is around 300MiB in size and contains 1 million words. As the distributed event processor is developed in Go, a similar baseline wordcount is also created in Go to eliminate differences between programming languages.

The baseline program is a simple go program that reads in a text file, and scans them through word by word incrementing each key in a hashmap. The general algorithm is below:

\begin{lstlisting}
func WordCount(s string) map[string]int {
	words := strings.Fields(s)
	m := make(map[string]int)
	for _, word := range words {
		m[word] += 1
	}
	return m
}
\end{lstlisting}

It is important to note that there is still an "upstream operator" that sends the words through a gRPC call to a downstream "operator" that increments in their respective map. This is done to emulate the network flow of the distributed stream data processor.

The baseline word count is then ran $10$ times and the execution time is taken then averaged to produce the final result. On a M1 Max processor, the average time of execution across 10 trials is \texttt{41.527422479} seconds.

\subsection{Distributed Stream Processor}
A similar process is used for benchmarking the distributed stream processor. The same 10 trials is run on top of the processor and an average is taken to get a comprehensive view on its average run times. The upstream operator is given the word file for it to be read and propagated throughout the system. One difference from this experiment is the amount of nodes chosen to exist in the chord network. The rationale is that with the increase in nodes, the more potential hopes data might need to take for it to travel to its responsible node. So as a test, the chord network is benchmarked on several node counts.

\section{Discussion}
It is clear that the chord protocol establishes an overhead, however it is surprising to see that the overhead is such a degree. This may be due to a couple of reasons. First being that gRPC might be saturating the internal networking bandwidth, however this is unlikely as the M1 Max processors support bandwidth of multi gigabit connections which should not be affected by one workload. Another possible issue is the amount of routing each request needs within the node, and this frequent lookups may have caused a hiccup. Finally, the node lookup process is not yet parallelized, which can create problems under massive workloads.

Some optimizations can be applied here. The first being caching an estimated key range where upstream operators can supply a watermark to the search request, which the target node will respond by acknowledgement to the gateway. This can allow the gateway to keep track and potentially enumerate a key range and if a certain key falls between that range, it can directly send the message across eliminating the hops in between nodes.

Another optimization that could be applied it to do network level optimizations of gRPC. Enforcing TCP stream reuse across all operations is a must as this decreases the times it takes to dial new connections to certain nodes. This approach however compromises the idea of having upstream nodes establishing connections to only gateway nodes, as it would require some connection between nodes within the Chord network to the upstream nodes as well.
